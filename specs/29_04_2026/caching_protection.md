# Caching Protection Spec - 29/04/2026

## Goal

Protect the Moralis-backed cache proxy from unbounded remote frontend behavior while keeping the external API Moralis-compatible.

Remote frontends can scroll, zoom, and request large ranges, but our API must enforce bounded page sizes, API-key budgets, and safe cache-miss behavior.

## Scope

Applies to external Moralis-compatible endpoints:

- `GET /api/v2.2/pairs/:pairAddress/ohlcv`
- `GET /token/mainnet/pairs/:pairAddress/ohlcv`

Does not necessarily apply to the internal chart route:

- `GET /api/charts/ohlcv`

The internal route can remain strict. The Moralis-compatible routes should be forgiving, page-safe, and budget-aware.

## Request Normalization

For every Moralis-compatible OHLCV request:

```ts
effectiveLimit = clamp(request.limit ?? 100, 1, 1000);
pageFrom = cursor?.fromDate ?? request.fromDate;
pageTo = min(request.toDate, pageFrom + effectiveLimit * timeframeMs);
```

Only serve `pageFrom -> pageTo`.

If `pageTo < request.toDate`, return a cursor:

```json
{
  "cursor": "...",
  "result": []
}
```

The cursor should encode the next safe `fromDate`.

## Oversized Requests

If a frontend sends:

```text
timeframe=1min
fromDate=2024-01-01
toDate=2026-04-29
limit=300
```

Do not reject because the full range is too large.

Instead serve:

```text
fromDate -> fromDate + 300 * 1min
```

and return a cursor for the next page.

## Hard Errors

Still return errors for truly invalid requests:

- Missing or invalid API key.
- Missing pair address.
- Unsupported timeframe.
- Invalid dates.
- `toDate <= fromDate`.
- Malformed cursor.

## CU Protection

Each API key should have protection limits:

- Max `limit`: `1000`.
- Max uncached Moralis pages per request: `1`.
- Max cache-miss requests per API key per minute.
- Max CU per API key per day.
- Optional max total requests per API key per minute.

If CU budget is exceeded:

Option A:
Return `429` with `Retry-After`.

Option B:
Return cached candles only with response headers:

```http
x-cache-source: cache-only
x-budget-exceeded: true
```

## Cache Behavior

For every request:

1. Normalize the requested page.
2. Check Redis response cache.
3. Check database candles.
4. Detect gaps only inside the safe page window.
5. If gaps are small and budget allows, fetch at most one Moralis page.
6. Store fetched candles.
7. Return Moralis-compatible response.

Never perform unbounded synchronous backfills for remote requests.

## Response Shape

Always return Moralis-compatible shape:

```json
{
  "cursor": null,
  "result": [
    {
      "timestamp": "2026-04-29T00:00:00.000Z",
      "open": 1,
      "high": 1,
      "low": 1,
      "close": 1,
      "volume": 100,
      "trades": 10
    }
  ]
}
```

## Observability Headers

Add response headers for debugging and partner support:

```http
x-cache-source: cache | cache+moralis | cache-only | partial
x-moralis-cu-used: 0 | 150 | ...
x-effective-limit: 300
x-page-from: 2026-04-01T00:00:00.000Z
x-page-to: 2026-04-01T05:00:00.000Z
```

## Persistent Usage Tracking

Production usage should be stored in database tables.

Local-memory/tunnel mode should persist usage to ignored local files:

- `.local-proxy-usage.json`
- `.local-proxy-usage.jsonl`

The summary file tracks totals. The JSONL file tracks every Moralis spend event.

## Security

API keys must be:

- Generated by admin endpoint.
- Stored hashed only.
- Required via `X-API-Key`.
- Rate-limited independently.
- Revocable.

Admin endpoints remain protected by `ADMIN_API_KEY`.

## Acceptance Criteria

- A 1-year `1min` request with `limit=300` returns 300 or fewer candles, not an error.
- The same request returns a cursor if more candles are available.
- No remote request can trigger more than one Moralis page fetch.
- Repeated identical requests are served from cache.
- Invalid API keys return `401`.
- Unsupported timeframes return `400`.
- CU usage is persisted and visible.
- Per-key CU limits prevent runaway usage.
- Internal `/api/charts/ohlcv` may remain strict.
